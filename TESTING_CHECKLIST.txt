HOSPITAL PROJECT TESTING CHECKLIST
==================================

Testing goal
------------
Before submission, testers should confirm that the patient portal, doctor portal, authentication, appointment lifecycle, waitlist, medical records, notifications, and database/API behavior all work correctly.


Main features to test
---------------------
1. Public pages
  - Patient welcome page
  - Doctor welcome page
  - About Us page
  - Navigation between patient and doctor welcome pages

2. Authentication
  - Patient signup
  - Doctor signup
  - Shared login for patient and doctor
  - Correct redirect based on role
  - Logout for both roles
  - Invalid login error handling
  - Duplicate email validation

3. Patient portal
  - Patient dashboard
  - Doctor list loading
  - Appointment booking
  - Time slot loading
  - Appointment list
  - Appointment details
  - Cancel appointment
  - Reschedule appointment
  - Join waitlist
  - Medical records list
  - Attachment download
  - Profile page
  - Notifications read and read-all

4. Doctor portal
  - Doctor dashboard stats
  - Schedule list
  - Appointment status changes
  - Cancel appointment
  - Reschedule appointment
  - Waitlist accept
  - Waitlist reject
  - Patient list
  - Patient timeline
  - Create medical record
  - Upload record attachment
  - Doctor settings
  - Working hours update
  - Day unavailable override
  - Notifications read and read-all

5. Backend API/security
  - Protected routes require login
  - Patient cannot access doctor-only endpoints
  - Doctor cannot access patient-only endpoints
  - JWT token is saved after login/signup
  - Invalid IDs and missing required fields return clear errors
  - Database collections are created and updated correctly

Scenario tests and expected outcomes
------------------------------------

SCENARIO 1: Backend health check
Steps:
  1. Start MongoDB.
  2. Start backend with npm run dev.
  3. Open http://localhost:5000/api/health.
Expected:
  - Response success is true.
  - Message says the Hospital API is running.
  - No backend crash.

SCENARIO 2: Frontend loads
Steps:
  1. Start frontend with npm run dev.
  2. Open http://localhost:5173.
Expected:
  - App opens without blank page.
  - Default route goes to patient welcome.
  - Browser console has no fatal errors.

SCENARIO 3: Patient signup
Steps:
  1. Open /signup.
  2. Fill full name, email, phone, password, confirm password.
  3. Submit.
Expected:
  - Account is created in MongoDB with role patient.
  - User is redirected to /patient/dashboard.
  - Token and active user are saved in localStorage.

SCENARIO 4: Doctor signup
Steps:
  1. Open /doctor/signup.
  2. Fill full name, email, phone, specialization, password, confirm password.
  3. Submit.
Expected:
  - Account is created in MongoDB with role doctor.
  - Doctor profile has specialization and department.
  - User is redirected to /doctor/dashboard.

SCENARIO 5: Doctor signup with custom specialization
Steps:
  1. Open /doctor/signup.
  2. Choose specialization Other.
  3. Enter a custom specialization.
  4. Submit.
Expected:
  - Custom specialization is required.
  - Account saves the custom specialization.
  - User is redirected to doctor dashboard.

SCENARIO 6: Duplicate email
Steps:
  1. Try to sign up with an email already used by another user.
Expected:
  - Signup fails.
  - Message says email is already registered.
  - No duplicate user is created.

SCENARIO 7: Password confirmation mismatch
Steps:
  1. Open patient or doctor signup.
  2. Enter different password and confirm password values.
  3. Submit.
Expected:
  - Form does not submit to dashboard.
  - Error says passwords do not match.

SCENARIO 8: Shared login as patient
Steps:
  1. Open /login.
  2. Enter patient email and password.
Expected:
  - Login succeeds.
  - User is redirected to /patient/dashboard.
  - Patient pages use patient token.

SCENARIO 9: Shared login as doctor
Steps:
  1. Open /login or /doctor/login.
  2. Enter doctor email and password.
Expected:
  - Login succeeds.
  - User is redirected to /doctor/dashboard.
  - Doctor pages use doctor token.

SCENARIO 10: Invalid login
Steps:
  1. Open /login.
  2. Enter wrong password.
Expected:

  - Error says invalid email or password.
  - User stays on login page.

  2. Open /patient/dashboard.
Expected:

Expected:

  - Doctor can see the appointment in schedule.

SCENARIO 30: Authorization protection
Steps:
  1. Login as patient.
  2. Try to access /doctor/dashboard.
  3. Login as doctor.
  4. Try patient-only API actions.
Expected:
  - Unauthorized role access is blocked by backend.
  - API returns forbidden/unauthorized error.
  - No protected data leaks between roles.

SCENARIO 31: Database verification
Steps:
  1. Open mongosh.
  2. Run:
       use hospital_management
       show collections
       db.users.find({}, { email: 1, role: 1, fullName: 1 }).pretty()
       db.appointments.find().pretty()
       db.medicalrecords.find().pretty()
       db.waitlistentries.find().pretty()
Expected:
  - Users have correct roles.
  - Appointments are linked to patient and doctor IDs.
  - Medical records are linked to appointment, patient, and doctor.
  - Waitlist entries reflect accepted/rejected/pending states.
